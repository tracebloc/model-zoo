# Embeddings (contrastive)

Train a text encoder with a self-supervised contrastive objective: map each input sentence to a dense embedding such that semantically similar texts land close together and dissimilar ones far apart. No labels are required — positives are built from the data itself (e.g. dropout-augmented views of the same sentence).

The model is a **plain text encoder**. It returns `last_hidden_state` shaped `(batch, seq_len, hidden)`; the training container mean-pools it (attention-mask aware) into one embedding per example and applies the contrastive loss. There is no classification, seq2seq, or language-model head.

## Start here

**Just validating the pipeline?** Use [`pytorch/simple_text_encoder.py`](pytorch/simple_text_encoder.py) — a small from-scratch transformer encoder that confirms the data, tokenizer, and pooling are wired up correctly.

To **fine-tune a pretrained** sentence encoder instead of training from scratch, use [`pytorch/minilm.py`](pytorch/minilm.py).

## Models

| Model | Params | When to pick |
|---|---|---|
| [`simple_text_encoder.py`](pytorch/simple_text_encoder.py) | ~10M | From-scratch baseline; pipeline validation and quick iteration |
| [`minilm.py`](pytorch/minilm.py) | ~22M | Fine-tune pretrained all-MiniLM-L6-v2 (HuggingFace) instead of training from scratch |

`simple_text_encoder.py` is a plain `nn.Module` (no pretrained weights, no in-file PEFT) returning a HuggingFace `BaseModelOutput` whose `last_hidden_state` is shaped `(batch, seq_len, hidden)`. `minilm.py` is a HuggingFace `AutoModel` (the bare 6-layer BERT encoder) returned directly — it ships pretrained weights and exposes `.config`. LoRA, if wanted, is selected in the training plan — never bundled into the model file (the package check rejects in-file PEFT).

## Dataset expectations

- **Input**: plain text lines (one example per line). Positives for the contrastive objective are generated by the training container; the data carries no labels.
- **Output contract**: the model returns `last_hidden_state`; the container does the (mask-aware) mean-pooling. The model performs **no** pooling, projection, label-shifting, or target building.
- **Batch size**: 16.

## Tokenizer

The tokenizer is the federation's single source of truth, distributed to every client (issue #805). How it is supplied depends on whether the model is HuggingFace or custom:

- **`minilm.py` (HuggingFace)** declares a `tokenizer_id`; the client loads the matching all-MiniLM-L6-v2 WordPiece tokenizer from the Hub (it already defines `[PAD]`, `[CLS]`, and `[SEP]`). Upload it with no tokenizer argument:

  ```python
  user.upload_model("model_zoo/embeddings/pytorch/minilm.py")
  ```

- **`simple_text_encoder.py` (custom)** uses [`pytorch/embeddings_tokenizer.json`](pytorch/embeddings_tokenizer.json) — a WordPiece tokenizer over the standard 30522-token bert-base-uncased vocabulary, `[CLS]…[SEP]` auto-wrapping, and `[PAD]` (id 0) as the pad token. `[PAD]` is the only special token embeddings requires (there is no masking, so no `[MASK]` is needed). Its max token id (30521) fits the model's embedding table.

  This file is deliberately **not** named `tokenizer.json`: the SDK auto-detects a sibling `tokenizer.json` and ships it for *every* model in the directory, which would override `minilm.py`'s `tokenizer_id`. Because of the non-default name it is not auto-detected, so pass it explicitly:

  ```python
  user.upload_model(
      "model_zoo/embeddings/pytorch/simple_text_encoder.py",
      tokenizer="model_zoo/embeddings/pytorch/embeddings_tokenizer.json",
  )
  ```
